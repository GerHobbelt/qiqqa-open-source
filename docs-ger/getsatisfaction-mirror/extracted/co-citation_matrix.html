    I suspect that you are more familiar with this than I :), so I'm flexible

    ... but here are my "rough" thoughts... (disclaimer... I spent about 15 minutes with this, so there maybe some errors in thinking)

    I would like to investigate Title X Title, and Author X Author using UCINET.

    The format used for authorsXauthor would need to be weighted, titleXtitle would simply be "linked or not". I suppose I could work with a matrix coded as a tab delimited file... the column/row names would most likely need to be coded though.

    So, something like

    For titles...
    bibtexkey1 /TAB bibtexkey2 /TAB bibtexkey3
    bibtexkey1 /TAB 0 /TAB 1 /TAB 0
    bibtexkey2 /TAB 1 /TAB 0 /TAB 1
    bibtexkey3 /TAB 1 /TAB 0 /TAB 0

    For authors, you'd have to generate a unique, but readable key for each author
    f.m.last /TAB f.m.last /TAB f.m.last
    f.m.last /TAB 0 /TAB 1 /TAB 0
    f.m.last /TAB 1 /TAB 0 /TAB 1
    f.m.last /TAB 1 /TAB 0 /TAB 0

    If you wanted to get "fancy"...The VNA format would allow for expression of node attributes, the DL format is simplest, as it only expresses the simple relationship (but has some sever limitation on node attributes (node "key's" cannot contain spaces, or special characters, and must be no more than 21 chars). VNA format would provide the greatest potential for added features - but would take some design thought as to what "made-sense". I suspect that you'd want to discuss amongst yourselves as to the best approach.

    For Titles use nodelist format in DL....
    dl
    n = 50
    labels embedded
    format = nodelist
    data:
    bibtexkey1 bibtexkey2 bibtexkey3 bibtexkey4 bibtexkey5
    bibtexkey2 bibtexkey1 bibtexkey5
    bibtexkey3 bibtexkey1 bibtexkey4 bibtexkey2
    etc...

    For Authors, use edge list weighted format
    dl
    n = 50
    labels embedded
    format = edgelist
    data:
    author1 author2 weight
    author1 author4 weight
    author2 author1 weight
    author3 author1 weight
    etc...

    PS
    Also - you may want to begin "expressing" the citation analysis structure you probably already have behind the scenes through the application interface.

    PPS
    Someone more familiar with the various NA software might have a better format suggestion - I guess, the format that is "most" universal AND "most expressive" would be the best.

    I've had a bit more time to look at this this morning.

    In regards to the citation matrix... I data file doesn't seem quite right... but I think any needed changes would be super simple to make.

    Though I can write a script to translate this into other formats (to excel, to network analysis software...a script I would provide back to anyone that wanted it)... but every entry in the database should have a line, even if it doesn’t reference another file or contain a pdf. Also a node count (number of records) would be nice as well (though I admit, not essential because it could be calculated from the updated data format).

    ...here is a summary

    1) Every entry in the database should have a line

    2) There shouldn't be any reason to ask about forward and backward (incoming outgoing) is there? To illustrate, here is a pseudo code description....processing should probably be something like this...

    * open file
    * print header information
    * print new line
    * print node count (number of records) as "n=COUNT"
    * for each entry in the datbase (pdf, or no pdf)
    * print new line
    * print bibtex key
    * for each reference in pdf
    * break if non-pdf
    * print {deliminator} (in this case, a comma)
    * print referenced bibtext key
    * end for
    * end for
    * close file

    The backward and forward look at the data is simply how we choose to view the data.

    Also – co-citation analyses can really only be done at an author level. Would it be easy to add an author citation export using the same format that you used with titles?

    Format is:
    author1,citedauthor1,citedauthor2,citedauthor3,
    author2,citedauthor1,citedauthor1,
    etc...
